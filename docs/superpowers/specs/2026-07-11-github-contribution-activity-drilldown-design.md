# GitHub 贡献活动联动设计

## 目标

将 GitHub 活动区域改为左侧紧凑月历、右侧 `Contribution activity` 的双栏布局。右侧默认展示所选月份的可靠汇总；点击日历日期后切换为当天贡献详情，让贡献数量能够下钻到仓库级别。

## 布局与交互

活动主卡使用约 55% / 45% 的左右双栏。左栏包含现有月份标题、上一月、`TODAY`、下一月、星期标题、日期网格和紧凑统计带。右栏拥有独立标题与内容区。

- 未选择日期时，右栏显示当前月份、月度总提交数、涉及仓库数量，以及各仓库的提交数。
- 点击有贡献的当月日期后，右栏显示完整日期、当天总贡献数、涉及仓库数量，以及各仓库当天提交数。
- 点击无贡献日期后，右栏显示该日期没有贡献。
- 再次点击当前已选日期，或切换月份，清除日期选择并恢复月度汇总。
- 选中日期具有可见选中态和 `aria-pressed` 状态；日期继续支持键盘操作和贡献提示。

## 数据与接口

现有 `contributionMonths[].counts` 和 `levels` 继续驱动日历。后端新增一个受现有 GitHub 缓存和错误边界约束的贡献详情接口，接收 `date=YYYY-MM-DD` 或 `month=YYYY-MM` 中的一种查询范围。

配置 GitHub Token 时，后端通过 GitHub GraphQL `contributionsCollection` 在指定起止时间内读取 `commitContributionsByRepository`，返回仓库名、仓库 URL、提交数、范围总数与范围类型。日期范围按本地日历日构造明确的 ISO 起止时间；月度范围从月初到下月月初。

无 Token、GraphQL 不可用或请求失败时，接口不得猜测仓库。日期模式使用已缓存月历中的当天总贡献数，月份模式使用已缓存月度总数，并返回 `detailsAvailable: false` 和安全提示。前端仍展示可靠总数，同时说明仓库明细需要 GitHub Token。

## 前端状态与并发

renderer 维护当前选中日期、活动详情缓存、加载状态和请求序号。选择日期时只更新右侧面板；相同范围的已成功结果从内存缓存读取。若用户快速切换日期，只有最新请求序号对应的响应可以更新界面，旧响应被忽略。

切换月份会清除日期选择并请求或读取该月份汇总。刷新 GitHub 数据后清空活动详情缓存，当前月份恢复月度视图，避免展示与新月历不一致的旧详情。

## 错误与空状态

- 日期无贡献：显示明确的零贡献状态，不发送仓库猜测。
- 无 Token：显示总数和需要 Token 才能获得仓库明细的提示。
- 网络或 GitHub 错误：保留当前可靠总数，显示可重试提示，不清空日历。
- 范围参数无效：后端返回 400，不向 GitHub 发出请求。

## 响应式与验证

宽屏使用 55% / 45% 双栏；中小屏改为上下排列，月历在上、活动详情在下。验证覆盖范围参数校验、GraphQL 映射、无 Token 降级、前端月度默认态、日期选择/取消、月份切换复位、旧请求隔离、无贡献状态和响应式结构。运行后端测试、renderer 测试和完整项目检查，并在真实 Electron 窗口验证宽屏与窄屏布局。
